# Government Policy Risk Assessment Engine

This is a **standalone subsystem** of the SchemeLens project. While the "Recommendation Engine" helps citizens find schemes, this "Risk Assessment Engine" operates completely separately to help the government identify flawed or dangerous policies.

## 1. What Other Risks Exist (Beyond Tax & Migration)?
If we analyze schemes, tax increases are just one type of risk. A robust algorithm should evaluate policies against these critical risk factors:

1. **Accessibility / Exclusion Risk:** Schemes with overly complex eligibility criteria (e.g., requiring 10 different documents or online-only portals) that unintentionally exclude the poorest, illiterate, or most vulnerable citizens.
2. **Bureaucratic Friction (Red Tape):** Schemes requiring approval from multiple government departments, leading to massive delays and a high risk of corruption/bribery.
3. **Market Distortion / Dependency Risk:** Handouts or subsidies that destroy local business competition or create long-term dependency without actually building citizens' skills or financial independence.
4. **Ecological / Environmental Risk:** Agricultural or industrial schemes that encourage practices leading to resource depletion (e.g., subsidies for water-heavy crops causing drought).
5. **Social Friction:** Policies that disproportionately benefit one specific demographic, potentially leading to social unrest or demographic shifts/migration.

## 2. How to Implement This Algorithmically
We will build a completely separate script (e.g., `government_risk_analyzer.py`) that does not interfere with the recommendation AI. We will use a **Multi-Factor Risk Scoring Algorithm** combining Natural Language Processing (NLP) and Rule-Based Logic.

### Step 1: Constraint & Entity Extraction
The algorithm reads a scheme's text and uses NLP (like the `spaCy` library) to extract structural data:
- How many eligibility conditions are there?
- How many documents are required?
- Does it mention financial keywords ("fee", "penalty", "mandatory contribution")?

### Step 2: Algorithmic Scoring (The Math)
The algorithm applies weights and rules to the extracted data to calculate sub-scores:
- **Exclusion Score:** `If condition_count > 5 -> Add +3 to Exclusion Risk.`
- **Red Tape Score:** `If department_count > 2 -> Add +4 to Red Tape Risk.`
- **Financial Risk:** `If "mandatory" AND ("tax" OR "fee") found -> Add +5 to Financial Risk.`

### Step 3: Zero-Shot AI Classification
For complex risks like "Environmental Damage" or "Social Friction", the algorithm passes the text to a Zero-Shot Classification model (a type of AI). The AI mathematically calculates the probability (0.0 to 1.0) that the scheme description matches the concept of "environmental damage".

### Step 4: The Final Weighted Risk Score
The algorithm calculates a final danger level:
`Total_Risk = (0.3 * Financial) + (0.2 * Exclusion) + (0.2 * Red Tape) + (0.3 * AI_Classification)`

## 3. Project Architecture (Separation of Concerns)
To ensure this doesn't mess with the recommendation code:
- **Database:** The script will read from `schemelens.db` but will write its results to a completely separate table called `government_risk_analysis`.
- **Backend:** We will add a completely separate API endpoint (e.g., `GET /api/gov/risky-schemes`).
- **Frontend:** There will be a separate "Government Login/Portal" page. Regular citizens will never see this risk data; it is exclusively an analytical dashboard for policy-makers.
